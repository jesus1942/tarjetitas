Objetivo del curso

Enseñar a diseñar y construir un sistema real para el buffet escolar: alumno, buffet, administracion, backend, seguridad y futura billetera de creditos.

Modulo 1

Recorrido del curso

Base actual del proyecto
Paso 01

Entender el problema real

El objetivo no es hacer un QR lindo. El objetivo es evitar fraude, duplicados, demoras y carga manual en la entrega del almuerzo.

Paso 02

Separar responsabilidades

Alumno muestra credencial, buffet valida, administracion configura y el backend decide. Esa separacion ordena todo el sistema.

Paso 03

Diseñar antes de programar

Primero se definieron arquitectura, flujos, API y modelo de datos. Eso evita improvisar reglas delicadas en el medio del desarrollo.

Paso 04

Prototipar interfaces operativas

Se construyeron pantallas para alumno, buffet y admin para validar UX, pero todavia no estan conectadas a un backend real.

Modulo 2

Roadmap pragmático

Lo que ya esta y lo que sigue

Etapa 0. Definición

Se cerró el alcance inicial, se documentaron flujos, arquitectura, modelo de datos, reglas de negocio y contrato de API.

Etapa 1. MVP técnico

Lo siguiente es tener circuito completo funcionando: login, credencial dinamica, validacion online, una entrega por dia y panel admin basico.

Etapa 2. Piloto real

Cuando el MVP cierre, toca probar en operacion real con importacion de alumnos, reportes diarios, bloqueos y ajustes de UX.

Etapa 3. Evolución

Despues se puede crecer a dispositivos, auditoria más fuerte, modo offline controlado y wallet de creditos con pagos.

Estado actual del repositorio

Hoy el proyecto ya tiene una PWA funcional para imprimir tarjetitas fisicas y, en paralelo, una base documental solida para el sistema digital. Las apps de alumno, buffet y admin existen como prototipos HTML, pero aun no estan integradas ni operando contra una API real.

Modulo 3

Arquitectura del sistema

Principio central

App del alumno

Muestra estado del dia, identidad y credencial temporal. No debe decidir por su cuenta si el almuerzo fue consumido o no.

App del buffet

Escanea, valida y responde rapido. Tiene que ser la interfaz mas agil del sistema porque trabaja con fila y tiempo real.

Panel admin

Gestiona alumnos, permisos, operadores, reportes, incidencias y configuraciones sin mezclarse con el retiro diario.

Backend central

Autentica, emite credenciales, valida consumos, aplica reglas antifraude y deja auditoria. Es la fuente de verdad.

Una entrega por alumno por fecha y servicio.
Credencial con vigencia corta.
Registro de operador, hora y resultado.

Flujo minimo ideal

Alumno abre la app, recibe credencial temporal, el buffet la valida y el backend responde aprobado o rechazo con motivo.

Alumno -> credencial temporal -> Buffet -> API -> validacion -> auditoria -> respuesta
Modulo 4

Como estudiar este proyecto

Didáctica aplicada

Primero el contexto

No empezar por código. Empezar por actores, dolor operativo, fraude posible y reglas de negocio.

Después el modelo

Diseñar datos, API y estados posibles antes de tocar integración o autenticación.

Recién ahí la implementación

Construir interfaz y backend con decisiones ya justificadas, no por intuición aislada.

Preguntas guía para clase

¿Por qué el alumno no puede ser la fuente de verdad?
¿Qué pasa si alguien reutiliza una captura de pantalla?
¿Qué se rompe si no guardamos auditoría?
¿Qué cambia entre un permiso diario y una billetera de créditos?
Modulo 5

Evolución futura: créditos y pagos

Diseñar sin rehacer todo

Modo beneficio escolar

El sistema decide si el alumno puede retirar hoy. La regla es permiso, fecha y servicio.

Modo wallet prepaga

El sistema administra saldo, recargas, debitos y compras. La regla es dinero convertido a creditos y productos consumidos.

Recargas por Mercado Pago

El familiar paga, se registra la transaccion y se acredita saldo al alumno con trazabilidad y validacion del pago.

Tarjetas fisicas o NFC

Puede sumarse una tarjeta para identificar al alumno mas rapido, sin eliminar la opcion de QR o celular.

Reglas importantes para esa evolución

No guardar solo saldo actual: guardar movimientos.
La equivalencia pesos-creditos debe ser configurable.
Cada compra debe registrar la tasa aplicada en ese momento.
Toda recarga, debito, ajuste o reversa debe auditarse.
